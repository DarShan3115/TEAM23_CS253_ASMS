Critical vulnerabilities
CRITICAL
No authentication on academic-service
Django settings.py sets DEFAULT_PERMISSION_CLASSES = AllowAny and CORS_ALLOW_ALL_ORIGINS = True. Every endpoint (enroll students, grade updates, attendance, announcements) is completely open — anyone who knows the URL can call them without a token.
server/academic-service/academic/settings.py · core/views.py
CRITICAL
No authentication on productivity-service (Go)
All routes under /api/productivity/v1/ — tasks, discussions, submissions, grading — accept any x-user-id header without verifying a JWT. Any user can impersonate another by forging the header and read/delete/grade anyone's work.
server/productivity-service/main.go · all handlers
CRITICAL
Analytics reads wrong table — always returns 0% attendance
calculations.py queries an attendance_logs table, but the actual schema (init.sql) only has an attendance table. models.py even creates this fake table on startup via Base.metadata.create_all(). Every student will show 0% attendance and get flagged as "at-risk" incorrectly.
server/analytics-service/models.py · calculations.py
CRITICAL
Hardcoded secrets in docker-compose.yml
DB password (password), Django SECRET_KEY (django-insecure-key), and JWT placeholder strings are hardcoded directly in the compose file, not from .env. These would be committed to git and exposed.
docker-compose.yml
CRITICAL
Student can enroll in any course without authorization check
CourseEnrollView and CourseEnrollByCodeView only validate x-user-id from a header (not a verified token). A student can set any user ID and enroll on behalf of anyone. Faculty/admin can also enroll themselves as students.
server/academic-service/core/views.py
Security warnings
WARN
OTP generated with Math.random() — not cryptographically secure
genOTP() uses Math.floor(100000 + Math.random() * 900000). Node.js Math.random() is not a CSPRNG — it should use crypto.randomInt(100000, 999999) instead.
server/auth-service/src/controllers/auth.controller.js · users.controller.js
WARN
OTPs stored in-memory only (Map) — lost on server restart
otpStore and resetOtpStore are plain JavaScript Maps in process memory. A crash or restart invalidates all pending OTPs mid-flow. Should use Redis or DB-backed storage.
server/auth-service/src/config/otpStore.js · auth.controller.js
WARN
CORS set to origin: true (reflects any origin) in auth-service
Comment says "safe behind proxy" but this allows any site to make credentialed requests to the auth service. Should whitelist known frontend origins explicitly.
server/auth-service/src/app.js
WARN
Rate limiter set to 1000 req/15min — effectively disabled
Comment says "increased for development flexibility" but this is too high for production and provides no real brute-force protection for login or OTP endpoints.
server/auth-service/src/app.js
WARN
JWT access token stored in localStorage — XSS-vulnerable
Frontend stores the token in localStorage which is accessible to any JavaScript on the page. Should store in an HttpOnly cookie (like the refresh token already does) or use a secure in-memory store.
client/src/store/authStore.js · client/src/services/api.js
WARN
No role verification when faculty grade or post announcements
UpdateFinalGradeView, CourseAnnouncementsView (POST), and GradeSubmission in Go don't check that the calling user is actually the instructor of that course. Any authenticated user could grade or announce.
academic-service/core/views.py · productivity-service/handlers/submission_handler.go
WARN
Avatar upload allows path-traversal via file extension
Multer uses the original file extension from file.originalname. If someone provides a filename like shell.php.jpg the mimetype check passes but the .jpg extension is preserved — could be dangerous on some server configs. Should strip extension and always append .jpg.
server/auth-service/src/routes/users.routes.js
Logic errors & bugs
BUG
Duplicate grading endpoints with different field names
GradeSubmission (submission_handler.go) accepts marks (int), while UpdateSubmissionGrade (assignment_management.go) accepts grade (float64). Both exist but only GradeSubmission is registered on a route. The schema has both marks (integer) and grade (numeric) columns — inconsistency.
productivity-service/handlers/submission_handler.go · assignment_management.go
BUG
Productivity service default DB fallback uses wrong credentials
The fallback DSN in main.go is user=admin password=admin but the actual DB user is postgres/password. If DATABASE_URL env var is missing, startup will fail.
server/productivity-service/main.go
BUG
AdminControlPanel route accessible before checking admin's own data loads
AcademicProgressPage route shows at /academic in sidebar but is not defined in App.jsx routes — navigating to it hits the * wildcard and redirects to the role home. Silent dead link.
client/src/App.jsx
BUG
Analytics calculate_subject_attendance checks for status == 'Present' (capital P)
But the DB attendance table stores values as 'present', 'absent', 'late' (lowercase, per schema CHECK constraint). Case mismatch means present count is always 0 even if the table were fixed.
server/analytics-service/calculations.py line: if l.status == 'Present'
BUG
Admin search query uses same param index for 3 columns
In listUsers: OR email ILIKE $${params.length + 1} reuses the same $N for first_name, last_name, and email. The SQL will bind the search term correctly but conceptually all three columns share the same placeholder index which only works by coincidence in pg when all three are the same value.
server/auth-service/src/controllers/admin.controller.js
What's working well
GOOD
Auth-service has solid brute-force protection: 5-attempt lockout, 15-min lock window, and admin unlock endpoint. Refresh token is stored in HttpOnly cookie with sameSite: strict.
GOOD
Password change flow is well-designed: OTP verification + current password check + strength regex + no-reuse rule, all enforced server-side.
GOOD
Discussion anonymity is enforced server-side in Go handler: faculty cannot post anonymously regardless of what the frontend sends.
GOOD
Forgot-password flow prevents email enumeration: always returns 200 with same message whether email exists or not.
GOOD
Frontend token-refresh interceptor handles concurrency correctly — queues parallel 401s and replays them all after one refresh, avoiding duplicate refresh storms.
